The integration is a direct API call and is not using Zapier, Make or a HubSpot widget. Upon submission, the JavaScript on the landing page posts the form data to an endpoint (Lambda function hosted behind API Gateway) and does not reach out to HubSpot's APIs from the browser to ensure security of the credentials. This API call triggers another API call to the HubSpot CRM v3 Contacts API to upsert the record, then the Karix WhatsApp Business API for sending the confirmation template and sends success response to the browser where Google Ads conversion event is triggered through GTM. It is more efficient than Zapier or Make integrations for this number of leads because both charge per task and do polling at certain intervals whereas our solution reaches out instantaneously without any risk of missing the SLA. In addition, we have full control over retries, latency and credentials being stored in one place. The disadvantage is maintainability of the integration but a developer would be responsible for maintaining the Lambda function anyway since of the volume of leads.

The most critical failing step is the deduplication of HubSpot which relies on emails but there are none collected in this form. The workaround labels the phone field as a unique value field and the Lambda function searches for a contact in the Contacts Search API by phone before making any creation attempt, if no one matches, it creates a contact using phone as a unique key and if someone does, it uses PATCH to update clinic preference, lead status and name on the existing contact instead of creating a duplicate. This will result in the overwriting of the name field of the first patient's contact in case two patients have entered their data via the form using the same phone number which happens often when the family phone is being used as an example; the phone, not the patient, is the unique entity.

WhatsApp SLA violation is possible when the template is approved by Karix but not Meta after any edit, in case of non-compliance with Karix rate limiting at a traffic surge or due to non-conformity with parallel processing in the Lambda function. Logging records the difference between the timestamps of request submission and the Karix response for each request and if this difference is above 90 seconds, a CloudWatch alert is triggered to Slack.
